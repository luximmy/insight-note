# InsightNote PRD (MVP)

## 1. 项目定位

一个 AI-Native 的智能研报助手。它能「读懂」PDF，并通过「生成式 UI」把文字转化为可交互的分析组件（图表等）。

## 2. MVP 北极星（一句话）

用户在 **5 分钟内**上传 **一份 PDF**，能通过对话得到 **基于文档的回答（真 RAG）**，并至少看到 **一种生成式 UI（至少一张趋势/数据类图表）**。

## 3. MVP 范围内

- **真 RAG**：见下文定义；回答须受检索到的文档片段约束，而非仅依赖模型闲聊。
- **对话**：流式输出、可连续多轮；与当前文档/当前会话绑定（见范围外：多用户与长期文档库可延后）。
- **生成式 UI（MVP 必达）**：至少 **1 个图表类**工具（如趋势图），AI 通过 Tool 返回结构化数据，前端用约定组件（如 Recharts）渲染。
- **极简工作台**：侧边栏布局——左侧文档/会话入口，右侧沉浸式对话区。

## 4. 「真 RAG」在 MVP 中的定义（可验收）

以下四条需同时满足，缺一不可：

1. **解析**：PDF → 可检索文本；**MVP 假设为文本型 PDF**（可复制文字的研报）。扫描件 / OCR **不纳入 MVP**。
2. **分块**：对解析文本做固定策略分块（如按段落或字符窗口 + overlap）；单块大小与 overlap 在实现层有明确常量或配置说明。
3. **向量与检索**：分块经 Embedding 写入**持久化向量库**；用户提问时对查询做 Embedding，**语义检索 top-k 片段**，再将片段注入生成流程。
4. **生成与可感知**：模型回答须**使用检索结果**；产品上至少有一种可感知体现（如：可展开的引用/来源列表、或开发与演示阶段可接受的日志/调试视图），以便证明「不是假 RAG」。

## 5. 明确不做（V1.1 及以后）

- 扫描 PDF / OCR、多格式（Word、网页剪藏等）。
- 多租户、账号体系、复杂权限。
- **财务对比表等表格类 Generative UI**（`ComparisonTable`）：作为 **V1.1**，不阻塞 MVP 交付；若提前完成图表与 RAG 可插队实现。
- 极致动效（如 Skeleton → Content 的 View Transition）：面试加分项，非 MVP 判据。

## 6. MVP 验收清单（勾选即过关）

1. **Grounding**：使用固定测试 PDF，提问「仅文档内数字/表述才能回答」的问题，连续多次回答合理；且能展示或导出「本次检索用到的文本块」佐证。
2. **拒答/边界**：对文档外问题，行为可演示（如明确无法从该文档得出），不出现明显幻觉捏造文内信息。
3. **生成式 UI**：对话中触发 ≥1 次图表工具，页面出现真实图表（非静态占位图）。

## 7. 技术面试叙事（非功能需求）

- **RSC / App Router 与流式生成**：体现对 Next.js 最新数据流与 UI 形态的理解。
- **自定义 Tool Use**：AI 触发前端可渲染的 React 组件，而非纯 Markdown。
